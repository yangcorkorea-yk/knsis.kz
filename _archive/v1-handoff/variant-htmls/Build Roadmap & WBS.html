01 Executive summary

What we are building, who it is for, what success looks like.

Product

A Kazakhstan-facing K-Beauty consultation platform (not a marketplace, not a booking engine). End users browse curated treatments and verified clinics, request a free consultation, and are handed off to a human manager via WhatsApp or Telegram. Managers convert leads, coordinate Korea-visit logistics (airport pickup, hotel, interpreter), and follow up with aftercare. No prices are displayed, ever.

Primary user journeys

  1. Discover & request — Onboarding → Home → Category → Treatment → Clinic → Consult form → Done
  2. Korea visit — KV landing → Plan form → Confirmation (visa info, pickup, hotel)
  3. Authenticated session — Phone OTP → permissions → My page → in-app chat with manager
  4. Manager workbench — Inbox → Lead detail → assign clinic → send templated message → log activity → mark done → solicit review

Out of scope for v1

  • Payments, deposits, escrow, refunds
  • Public price lists or any visible pricing field
  • Doctor-side login (clinics communicate via manager only in v1)
  • Native iOS / Android apps — we ship a PWA that feels native
!
Hard constraint. No price input, no price display, no estimated cost preview — anywhere. All quotes are delivered 1-on-1 via WhatsApp/Telegram by a licensed manager after triage. Audit this in code review and in the moderation pipeline (reviews are scanned for price mentions).

03 Responsive strategy

Mobile-first end-user, desktop-first admin. Tablet is a real breakpoint.

Mobile · < 768 px

Primary surface. Designs are pixel-anchored to 390 px. Centered on tablet/desktop inside a 440 px max-width column for the public app. Bottom tab bar, sticky header, full-bleed media, 44 px hit targets, safe-area padding.

Tablet · 768–1279 px

Public app: stays in mobile column with side decoration (gradient or muted artwork). Admin: hidden behind a "best on desktop" notice with a read-only inbox fallback so a manager can still triage a lead from an iPad.

Desktop · ≥ 1280 px

Admin is its own canvas — 220 px sidebar, fluid main, 300 px detail drawer. Public app remains a centered mobile column with a marketing aside on Home/KV (clinic spotlight, latest reviews, language switcher).

Breakpoint contract (Tailwind tokens)

sm
≥ 640 px — large mobile / small tablet portrait
md
≥ 768 px — tablet portrait threshold (public app gets aside decoration)
lg
≥ 1024 px — tablet landscape (admin shows full sidebar)
xl
≥ 1280 px — desktop admin layout (sidebar + table + drawer)
2xl
≥ 1536 px — admin gets extra column for activity feed

Per-screen targets

  • Mobile suite (20 screens incl. Onboarding, Home, Categories, Treatment, Clinics, Reviews, Form, Done, MyPage, Korea Visit, Login, Before/After, Notifications, Chat, Search, Settings): mobile-first, centered ≤ 440 px column on larger screens.
  • Admin suite (10 screens incl. Dashboard, Leads, Customers, Clinics, Reviews mod, Managers, Notif templates, Automation, Send log, B/A mgmt): desktop-first at ≥ 1280 px. Below lg serve a degraded "go to desktop" or read-only view.

09 Integrations

Three messaging providers behind one abstraction. Phone OTP. Storage. Analytics.

WhatsApp Business (Meta Cloud API)

  • Pre-approved message templates for outbound; freeform allowed within 24-h window after a customer reply.
  • Webhook ingestion at /api/webhooks/whatsapp; signature verification mandatory.
  • Per-template registration in Meta Business Manager — automated via API where possible.
  • Owner: BE WBS M6-04 · ~4 days incl. Meta verification.

Telegram Bot API

  • One bot per region; users initiate by deep-link from Done screen.
  • No template approval — much faster to ship and freer copy.
  • Owner: BE WBS M6-05 · ~2 days.

SMS (Mobizon for KZ, fallback)

  • Used only when user has no WA / TG identity, and for OTP.
  • Sender ID needs operator approval per country.

Other

  • Phone OTP: Mobizon for KZ +7; Twilio Verify for other origins.
  • Storage: Supabase Storage; private bucket; presigned PUT/GET.
  • Email: Resend; transactional only (no marketing in v1).
  • Analytics: PostHog; product events + funnels + session replay (with PII masking).
  • Errors: Sentry; source maps; release health.

10 i18n & content

Three first-class languages. KZ default. Catalogs are source-of-truth, DB rows hold per-locale JSON.
  • Static copy lives in /messages/{kz,ru,kr}.json under next-intl. Mirror the existing i18n.jsx tree.
  • Dynamic content (treatments, clinics, templates) stores localized fields as JSON columns: { kz: "...", ru: "...", kr: "..." }. Helper t(field, locale) with fallback chain kz → ru → kr.
  • URL strategy: /[locale] prefix; default kz; Accept-Language sniff on first visit only.
  • Pluralization: next-intl ICU plural rules; KZ uses Russian plural form set.
  • Fonts: Pretendard Variable covers Latin, Cyrillic, Hangul; system fallback for rare Kazakh extended glyphs (ә ң ғ қ ұ ү һ і) is a no-op on modern devices.
  • Translation workflow: KZ master written by product; RU + KR done by native reviewers each sprint. PR-blocking lint: no untranslated keys before merge.

12 Non-functional requirements

Performance, accessibility, security, observability — budgets the team will be held to.

Performance budgets (Moto G4, 3G Slow)

  • LCP < 2.5 s on Home
  • CLS < 0.1 page-wide
  • TBT < 200 ms
  • JS bundle per route ≤ 130 KB gzipped
  • Images via next/image, AVIF/WebP, lazy below the fold

Accessibility

  • WCAG 2.1 AA on the entire public flow
  • Color contrast ≥ 4.5:1 on body text (audit our rose-on-tint chips!)
  • Keyboard reachable everywhere including the bottom tab bar
  • Form fields use real <label>, error text wired with aria-describedby
  • Focus visible ring (don't kill outlines)

Security & privacy

  • PII (phone, email, photos) encrypted at rest (Supabase column encryption)
  • Photos in a private bucket, signed GET with 5-min TTL
  • Rate limit every mutating route; Turnstile on unauthenticated forms
  • Audit log for all admin mutations
  • Data-residency: Supabase EU region by default; OK to expand
  • "Delete account" actually deletes / tombstones with 30-day reversal

Observability

  • Sentry for FE + BE errors, release tagging
  • PostHog for funnels: Onboard → Home → Tx → Lead, Korea Visit conversion
  • Provider webhook delivery dashboards (admin-only)
  • Inngest run history per automation rule

13 QA & testing strategy

Light unit, heavy integration, narrow e2e. Don't test the framework.
  • Unit (Vitest): pure utilities, RBAC, plural rules, template renderer, phone normalization.
  • Integration: API route handlers behind a real Postgres (testcontainers). The bulk of confidence lives here.
  • E2E (Playwright): 5 critical journeys — sign in, browse + lead, KV plan, admin assign + send, review moderation. Runs on every PR.
  • Visual regression: Chromatic / Playwright screenshots on the 28 designed screens at 390/768/1280 px. Block deploy on diffs.
  • Manual: i18n sweep by native speakers each sprint; a11y manual once per phase.
