01 Executive summary

Why an MVP slice, what it ships, what it defers.

Why we slice MVP this way

The full v1 design depends on five external services that each carry their own onboarding, legal review, or vendor approval cycle: phone OTP provider (Mobizon / Twilio Verify), WhatsApp Business Account verification with Meta, Telegram Bot deployment, SMS contracts, and the automation rule engine that ties them all together. Together they add roughly 30% of total engineering effort and almost all of the launch risk.

The MVP defers every one of them. We ship the customer-facing surface, the lead-capture flow, the manager workbench, and the in-app chat — everything we need to start receiving and converting consult requests today — and we land the external channels as a follow-up milestone once contracts and vendor approvals are in hand.

Primary user journeys (MVP)

  1. Discover & request — Onboarding → Home → Category → Treatment → Clinic → Consult form → Done
  2. Korea visit — KV landing → Plan form → Confirmation
  3. In-app chat — guest or returning user opens chat thread, talks to manager
  4. Manager workbench — Inbox → Lead detail → assign clinic → reply in-app → mark done → solicit review
!
Hard constraints (unchanged from v1). (1) No price input, no price display, no estimated cost preview — anywhere. (2) No medical claims in UI copy — every treatment page carries a disclaimer. (3) PII encrypted at rest. Photos in a private bucket served by 5-minute signed URLs. (4) Audit log every admin mutation with actor / before / after.

02 MVP scope · what's in / what's out

A single source of truth for the cut. Treat this as the contract.

✓ IN MVP

  • 22 mobile/admin screens, all 3 languages
  • Guest sessions (cookie-based, no login wall)
  • Staff login (email + password, seeded)
  • Consult form (name, phone, email, treatments, region, photos, message)
  • Lead lands in admin inbox in real time
  • In-app chat between customer and manager (Supabase Realtime)
  • In-app notifications inbox (server-pushed rows)
  • Korea-visit landing + plan form + confirmation
  • Admin: Dashboard · Leads · Customers · Clinics · Reviews · Managers · B&A
  • Review moderation with price-keyword scan
  • Before/After gallery with consent capture
  • i18n KZ default · RU · KR · sticky language pick
  • PWA — installable on Android Chrome
  • Sentry, PostHog, audit log, rate limits

✗ OUT (deferred to post-MVP)

  • Phone OTP login (Mobizon / Twilio Verify)
  • WhatsApp Cloud API adapter (Meta verification needed)
  • Telegram Bot API adapter
  • SMS adapter (Mobizon / SMSC KZ)
  • Email adapter for outbound campaigns (Resend) — basic transactional only in MVP
  • Template editor (per-channel × per-language)
  • Automation rule engine (trigger → delay → channel → template)
  • Inngest workers for outbound queues + retries
  • Send log UI with delivery status
  • Web push notifications (VAPID)
  • Apple / Google social login
  • Doctor-side login (clinics still talk to managers only)
  • Native iOS / Android apps
  • Payments / deposits / escrow (forever out of scope)
i
Schema is forward-compatible. The MVP Channel enum keeps the wa · tg · sms · email variants alongside inapp. The MVP only writes inapp and email rows, but the columns are there so the post-MVP rollout doesn't need a migration. Same applies to Message.providerId, Lead.channelPref, and the Template / AutomationRule tables — created but unused until M-Post.
